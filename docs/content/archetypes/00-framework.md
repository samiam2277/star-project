# 36 天命原型库 · 分类框架

> 对应 PRD §5.0.2 · v0.2 产品骨架
> 状态:**框架设计稿 v1**,待用户审阅敲定后启动命名 + 文案工程

---

## 1. 设计目标

把本命盘核心要素(太阳 × 月亮 × 上升 + 北交 + 凯龙)映射到 **36 个稳定的"天命原型"**,每个原型有独立的中英双语 2-3 字名(如"深潜者 Deep Diver"),作为产品中所有模块的公共上下文(`primary_archetype`)。

3 大设计约束(来自 PRD §5.0.2):

| 约束 | 含义 |
|---|---|
| **3×12 结构** | 网格化可视化(主题色 × 12 卡片),便于在分享卡、SEO 列表页、Onboarding 选择器中陈列 |
| **算法不暴露** | 用户只看"原型名 + 4 段文案",不解释怎么算出来的(类比 MBTI) |
| **降级保底** | 时间不准 / 完全无时间时,也能给出原型,不留空白 |

## 2. 顶层结构:3 阵营 × 12 原型

### 2.1 三阵营定义

阵营 = 该原型的**核心生命驱动力**。三阵营对应荣格的"三大灵性张力"(创造/关系/转化),映射到占星上是火土风水四元素 + 三模式的合力。

| 阵营 | 中文名 | English | 核心驱动 | 行星权重 | 主色调 |
|---|---|---|---|---|---|
| C 阵营 | **创造系** | **Creation** | "我要把内在带到世界" — 表达、塑造、留下印记 | 太阳、火星、5宫 | 暖琥珀 `#FFB347` |
| L 阵营 | **链接系** | **Connection** | "我通过关系成为自己" — 共鸣、桥接、调和 | 月亮、金星、7宫 | 玫瑰金 `#E8A1A1` |
| T 阵营 | **转化系** | **Transformation** | "我向深处去,再带回光" — 蜕变、洞察、疗愈 | 冥王、海王、8/12宫 | 深紫黛 `#6B5B95` |

**视觉应用:**
- 3 阵营对应 3 套色彩 token(`--archetype-creation-*` / `--archetype-connection-*` / `--archetype-transformation-*`)
- 分享卡背景渐变以阵营色为基,装饰金线统一(品牌一致性)
- 36 原型网格页(`/learn/archetypes`)用 3 行 × 12 列的网格,行间留白 + 阵营标题

### 2.2 每阵营 12 个原型 — 子主题轴

每个阵营内部用 12 个子主题区分,子主题不直接 = 12 星座,而是该阵营内的"功能侧重"。这样命名不会显得"白羊型创造者""金牛型创造者"这种乏味结构,而是有独立性格的 12 个角色。

## 3. 输入要素 → 原型 ID 的映射逻辑

### 3.1 输入信号优先级

```
                  ┌──────────────────┐
                  │ 1. 太阳元素 (Sun)  │ → 决定阵营倾向(粗筛 1/3)
                  └────────┬─────────┘
                           │
                  ┌────────▼─────────┐
                  │ 2. 月亮 + 火星     │ → 决定能量颜色(进一步缩到 4 个候选)
                  └────────┬─────────┘
                           │
                  ┌────────▼─────────┐
                  │ 3. 上升 (Asc)     │ → 决定"对外侧面",落到 2 个候选
                  └────────┬─────────┘
                           │
                  ┌────────▼─────────┐
                  │ 4. 北交 + 凯龙    │ → 最终确认 1 个原型 ID
                  └──────────────────┘
```

### 3.2 阵营分配规则(粗筛)

| 太阳元素 | 月亮元素 | 火星元素 | 阵营倾向 |
|---|---|---|---|
| 火 | 火/风 | 任意 | C 创造系 |
| 火 | 水/土 | 任意 | C 或 L(看上升) |
| 风 | 火/风 | 任意 | C 创造系 |
| 风 | 水/土 | 任意 | L 链接系 |
| 土 | 火/风 | 火/风 | C 创造系 |
| 土 | 土/水 | 任意 | L 链接系 |
| 水 | 任意 | 水/土 | T 转化系 |
| 水 | 火/风 | 火/风 | L 链接系 |

> 这是 v1 的**简化规则表**。正式实现时需要查 36 棵决策树,每棵决策树对应一个原型 ID 的"识别条件集合"(task #6 产出)。当多原型条件同时命中时,取**条件匹配数最高**的;同分时取**优先级权重最高**的(由占星师人工标注)。

### 3.3 在阵营内细分到 12 子主题

子主题轴的输入信号:

| 信号源 | 提取方式 | 影响 |
|---|---|---|
| 月亮宫位 | 1-12 宫 | 决定"内在功能定位"(如月3宫 → 编织/拓荒,月8宫 → 深潜/破茧) |
| 上升星座 | 12 星座 | 决定"对外形象"(如上升射手 → 远征/远眺/远观) |
| 北交宫位 | 1-12 宫 | 修饰"灵魂方向"侧重(影响进化路径文案) |
| 凯龙宫位 | 1-12 宫 | 修饰"阴影区"主题(影响阴影文案) |

具体的"宫位 → 12 子主题映射表"在 task #6 详写,这里仅给出关键启发式:

- **月亮+上升的"行动模式"** = 子主题的主特征
- **北交+凯龙** = 文案的色彩,但**不**改变原型 ID(原型是稳定身份,流年才动)

### 3.4 降级策略(出生时间不确定)

按 PRD §5.0.2 末段的降级规则:

| 数据完整度 | 落点策略 | 候选范围 |
|---|---|---|
| 精确到分钟 | 完整 36 原型决策树 | 1/36 |
| 上午/下午(已知半天) | 跳过上升,只用太阳+月亮+北交 | 缩到 ~12-18 个候选,从中随机锁定一个稳定的 |
| 完全不知道 | 仅用太阳+月亮 | 12×12=144 → 压缩到 36 子集,每组合落到一个**简化原型**(共 36 中的 ~20 个最稳定的) |

降级原型在 UI 上不打"降级"标签(避免用户负面感),仅在内部 metadata 标 `confidence: low|mid|high`,后续 AI 解读时会更谨慎(避免说"你的上升告诉我们..."这种依赖未知数据的话)。

### 3.5 决策树的数据契约

```typescript
// docs/content/archetypes/archetypes.json 的预期 schema
interface ArchetypeRule {
  id: string;                    // 'C01' | 'L05' | 'T12' ...
  faction: 'C' | 'L' | 'T';
  subThemeIndex: number;         // 1-12
  nameZh: string;                // '深潜者'
  nameEn: string;                // 'Deep Diver'

  conditions: {                  // 多条件,全部满足 = 命中
    sunSign?: ZodiacSign[];      // 可选,如 ['Scorpio', 'Pisces', 'Cancer']
    moonSign?: ZodiacSign[];
    moonHouse?: number[];        // 1-12
    ascSign?: ZodiacSign[];
    northNodeHouse?: number[];
    chironHouse?: number[];
    // ...其他可选触发器
  };

  priorityWeight: number;        // 0-100, 同时命中多原型时按此排序
  fallbackOnly?: boolean;        // 降级原型(无精确时间时可触发)
}
```

每个原型规则中的条件用 AND 组合,多原型同时命中时按 `priorityWeight` 降序选一个。无任何规则命中时(理论上不会发生),回落到该太阳元素的"默认原型"(每个元素一个,共 4 个保底)。

## 4. 命名风格指南

### 4.1 中文命名规则

- 长度: **2-3 字**(限 4 字仅在不可替代时,如"远眺者"3 字,可考虑"远眺"2 字看是否成立)
- 结构: **动作/状态 + "者/人/师"** 后缀为主(深潜+者、织梦+人、雕琢+师)
- 避免: 政治敏感词、宗教专用词(避免"使徒""菩萨")、生僻字
- 韵感: 同阵营 12 原型读起来要有节奏感,可念出来不拗口

### 4.2 英文命名规则

- 形式: **2 词为主**(Deep Diver、Light Bearer),少量 1 词(Alchemist、Oracle)、极少 3 词(暂无)
- 避免: 商标已注册词(需 task #1 中跑 USPTO/欧盟商标局盲检)
- 双语**不互译**:中文叫"深潜者",英文不一定叫"Deep Diver",可叫"Abyssal"或其他更英语化的命名(但 v1 框架我先用同义对应,正式定稿允许调整)

### 4.3 文化盲检

- 中文圈测试: 是否有方言/网络梗负面联想(如"打工人""躺平")
- 英语圈测试: 是否有黑人/原住民文化挪用风险(如"Shaman""Witch Doctor")
- 海外华人圈(主目标受众): 中英两端都能用,不能只对一边友好

## 6. 验收标准

本框架定稿需通过以下检查(用户审阅时勾选):

- [ ] 3 阵营的命名(创造/链接/转化)是否准确表达占星上的能量类型
- [ ] 12 子主题在每阵营内是否有足够区分度(读 36 个名字,不会觉得"这俩好像")
- [ ] 阵营色彩 token(琥珀/玫瑰金/紫黛)是否符合品牌"现代神秘主义"调性
- [ ] 降级策略是否足以处理"完全不知道出生时间"的用户(占预期用户 ~20%)
- [ ] 与 PRD §5.0.3 公共上下文的接口字段是否齐全
- [ ] 5 大模块的接口数据契约是否能直接对接到现有 Design Spec
